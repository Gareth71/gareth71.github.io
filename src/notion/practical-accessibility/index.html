  1. The four principles underlying WCAG (P.O.U.R)
    • Perceivable¬†means that users must be able to perceive Web content in some way, using one or more of their senses (it can‚Äôt be invisible to all of their senses)
      • 1.1 ‚Äì Text Alternatives:¬†Provide text alternatives for any non-text content so that it can be changed into other forms people need, such as large print, braille, speech, symbols or simpler language.
      • 1.2 ‚Äì Time-based Media:¬†Provide alternatives for time-based media.
      • 1.3 ‚Äì Adaptable:¬†Create content that can be presented in different ways (for example simpler layout) without losing information or structure.
      • 1.4 ‚Äì Distinguishable:¬†Make it easier for users to see and hear content including separating foreground from background.
    • Operable¬†means that users must be able to control UI elements (e.g. buttons must be activated in some way ‚Äî mouse, keyboard, voice command, etc.). The user interface cannot require interaction that a user cannot perform.
      • 2.1 ‚Äì Keyboard Accessible:¬†Make all functionality available from a keyboard.
      • 2.2 ‚Äì Enough Time:¬†Provide users enough time to read and use content.
      • 2.3 ‚Äì Seizures and Physical Reactions:¬†Do not design content in a way that is known to cause seizures or physical reactions.
      • 2.4 ‚Äì Navigable:¬†Provide ways to help users navigate, find content, and determine where they are.
      • 2.5 ‚Äì Input Modalities:¬†Make it easier for users to operate functionality through various inputs beyond keyboard.
    • Understandable¬†means that the content must be understandable to its users.
      • 3.1 ‚Äì Readable:¬†Make text content readable and understandable.
      • 3.2 ‚Äì Predictable:¬†Make Web pages appear and operate in predictable ways.
      • 3.3 ‚Äì Input Assistance:¬†Help users avoid and correct mistakes.
    • Robust¬†means that the content must be developed using well-adopted web standards that will work across different browsers, now and in the future.
      • 4.1 ‚Äì Compatible:¬†Maximize compatibility with current and future user agents, including assistive technologies.

Relevant WCAG

    programmatically determined or are available in text.‚Äù

    Must programatically define areas

  • 2.4.1 Bypass Blocks - Level A

    ‚ÄúA mechanism is available to bypass blocks of content that are repeated on multiple Web
    pages.‚Äù

    By programatically defining areas user can skip blocks

8 ARIA landmark types
  • banner
    • <header>
    • Contains global, site-oriented content repeated across pages of a website.
    • Only 1 per website,
    • should be placed outside other regions to get its role
      • not page specific content
  • navigation
    • <nav>
    • Contains a collection of navigational elements (e.g. links) for navigating within a website.
    • Can have multiple navs but if so must name them with aria-label or aria-labelledby
  • main
    • <main>
    • Designates the primary content of the page.
    • There should be only one main landmark.
    • should be placed outside other regions
  • contentinfo
    • <footer>
    • Contains information about the parent document.
    • typically has the same infor across pages of website
    • should be placed outside other regions
  • complementary
    • <aside>
    • Contains secondary information that is complementary to the main content. It should be related to the main content, but should also be easily understood without it.
      • eg list of complementary articles in a blog post.
    • should be sibling or child of main
    • requires an accessible name with aria-label or preferably aria-labelledby linked to a heading
  • form
    • <form>
    • to be a landmark it needs an accessible name with aria-label or aria-labelledby
    • don‚Äôt name too many forms only important ones
  • search
    • no HTML element add role=‚Äùsearch‚Äù to form or div
    • Represents a collection of elements that, combined, create a search facility.
  • region
    • <section> or <div role="region">
    • A generic region that represents a perceivable section containing content that is relevant to a specific purpose and sufficiently important that users will likely want to be able to navigate to the section easily and to have it listed in a summary of the page.
    • requires an accessible name with aria-label or preferably aria-labelledby linked to a heading

Should you choose a link or a button?
  • Does the element take the user to another page or a section within the page?

    If the URL changes when the element is clicked, then what you want is a link.

  • Does the element change something on the current page?

    If the answer is yes, you probably want a button.

  • Can you right-click the element to open it in a new Window?

    If not, then it's probably not a link.

  • Does the element do anything without JavaScript?

C15 Accessible Images: providing effective alt text for images

Why it is important

  • Alt text is important for people with little or no vision who use screen readers or other TTS assistive technologies to browse the Web and who rely on text to understand a page.
  • It is also important for people browsing with speech recognition. When an image doesn't load, these users can use the image's alt text in the voice command to trigger the action that the image may represent. (More on this later.)
  • Alt text is useful for SEO. Search engines are better able to index images when they have text alternatives.
  • It is also important for users on slow networks who have turned images off to save data, or for whom images simply won't load because their network is slow.
